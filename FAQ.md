## Topic 1：MCU 启动基础知识

### Q1：从上电到 main() 执行之间发生了什么？

**答：** 整个过程称为**启动流程（Boot Sequence）**，大致分以下几步：

1. **上电复位**：CPU 从复位向量（Reset Vector）取出第一条指令地址。ARM Cortex-M 系列从 `0x00000000` 读取堆栈指针（MSP），从 `0x00000004` 读取复位处理函数地址。
2. **系统时钟初始化**：芯片内部 PLL 开始工作，从默认的内部振荡器切换到外部晶振或更高频时钟。这一步通常在 `SystemInit()` 中完成（由启动文件或厂商库调用）。
3. **段初始化**：启动代码将 `.data` 段从 Flash 复制到 SRAM，并将 `.bss` 段清零。
4. **调用 C 运行时库**：初始化全局 C++ 构造函数（如果有）、设置堆栈边界等。
5. **跳转至 main()**：调用 `main()` 函数，应用代码开始执行。

以 STM32 为例，启动文件 `startup_stm32xxxx.s` 中的 Reset_Handler 伪代码如下：

```
Reset_Handler:
  LDR   R0, =SystemInit
  BLX   R0          ; 调用 SystemInit() 配置时钟
  LDR   R0, =__main  ; 调用 C 库初始化（段复制、BSS 清零）
  BX    R0          ; 最终跳转到 main()
```

**TL;DR：** 上电后 CPU 从复位向量取指 → 初始化时钟 → 复制/清零内存段 → 跳转到 main()。

### Q2：为什么 MCU 需要启动文件（.s 文件）？

**答：** 启动文件是用汇编编写的**硬件初始化胶水代码**，其必要性在于：

- **C 语言无法直接处理中断向量表**。中断向量表需要放在固定的起始地址，汇编可以精确控制每个入口地址的放置。
- **在 C 环境初始化之前完成必要配置**。例如设置堆栈指针 SP、配置系统时钟、初始化内存等。这些操作必须在任何 C 代码执行之前完成。
- **处理异常和中断的默认入口**。启动文件为所有中断提供了默认的 Weak Handler，开发者只需重定义同名函数即可。
- **调用 C 运行时初始化**。如前面提到的 `.data`/`.bss` 段复制清零，C 库初始化等。

如果没有启动文件，编译器不知道中断向量放哪里、堆栈指针初始值是多少、谁来调用 main()。

以 Cortex-M 为例，向量表的前两项必须是：

| 偏移 | 内容             | 示例值        |
|------|------------------|---------------|
| 0x00 | 初始 MSP 值      | 0x20008000    |
| 0x04 | Reset_Handler 地址 | 0x08000145    |

**TL;DR：** 启动文件是汇编写的"摆渡人"，负责把硬件从复位状态安全送到 C 语言环境。

### Q3：什么是 .data 和 .bss 段？为什么需要复制和清零？

**答：** 这是 C 语言编译产物中的两个关键内存段。

| 段名      | 内容                          | 存储位置         | 运行时需要         |
|-----------|-------------------------------|------------------|-------------------|
| `.data`   | 已初始化的全局/静态变量        | Flash（持久保存） | SRAM（可读写）     |
| `.bss`    | 未初始化或初始化为 0 的变量    | 不占用 Flash     | SRAM（可读写）     |
| `.text`   | 程序代码（指令）               | Flash            | Flash（只读）      |

**.data 复制的原因：** 变量需要可读写（Flash 是只读的），所以启动代码必须把 `.data` 从 Flash 复制到 SRAM 中。

**.bss 清零的原因：** C 标准要求未初始化的全局变量值为 0。Flash 中不保存 .bss 内容，所以启动代码必须在 SRAM 中将其清零。

如果忘记复制/清零，后果：

```c
int counter = 100;  // 位于 .data，未复制则为 Flash 中的值，无法写入！
int flag;           // 位于 .bss，未清零则为 SRAM 上电随机值！
```

**TL;DR：** `.data` 从 Flash 复制到 SRAM 是为了让变量可读写，`.bss` 清零是为了满足 C 标准"未初始化变量为 0"的要求。

### Q4：什么是链接脚本（Linker Script），为什么不能忽略它？

**答：** 链接脚本告诉链接器**如何将各个段安排到目标芯片的物理地址空间**。

它的核心作用：

- **指定 Flash 和 SRAM 的起始地址和大小**：不同的 MCU 有不同的内存映射，例如 STM32F103C8T6 的 Flash 在 `0x08000000`，SRAM 在 `0x20000000`。
- **定义段的放置规则**：`.text` 放 Flash，`.data` 的 LMA（加载地址）在 Flash、VMA（运行时地址）在 SRAM。
- **导出符号供启动文件使用**：如 `_sdata`、`_edata`、`_sbss`、`_ebss`、`_estack` 等。

忽略链接脚本的典型后果：

```c
// 链接脚本错误或缺失
#define ARRAY_SIZE 65536
uint8_t big_buffer[ARRAY_SIZE];  // 如果 SRAM 只有 20KB，编译器不知道，照常链接！
// 运行后程序直接跑飞 —— 访问了不存在的内存地址。
```

错误配置示例（芯片为 64KB Flash / 20KB SRAM）：

```
FLASH  (rx)  : ORIGIN = 0x08000000, LENGTH = 64K
RAM    (rw)  : ORIGIN = 0x20000000, LENGTH = 20K
// 如果将 LENGTH 错写为 256K，链接器不会报错，运行时必定崩溃。
```

**TL;DR：** 链接脚本是代码到芯片的"地址映射图"，错配或缺失会导致程序不工作甚至烧坏硬件。

### Q5：我的变量在运行时初始值不对，为什么？

**答：** 可能原因按出现频率排列：

**1. 启动文件没有正确复制 .data / 清零 .bss**

部分 IDE 或 CubeMX 生成的代码中，如果禁用或修改了启动文件的段初始化，变量将保持复位后的随机值。

**2. 变量定义在了错误的位置**

```c
// 在函数内部定义的局部变量
void func(void) {
    int value = 100;  // 这由编译器生成的代码初始化，不是启动文件负责
}

// 全局变量
int value = 100;      // 由启动文件复制 .data 段初始化
```

**3. 链接脚本中的段地址或长度配置错误**

`.data` 的 LMA 地址必须指向 Flash 中的正确位置，VMA 必须指向 SRAM 中足够大的区域。

**4. 变量被优化掉了**

使用 `volatile` 或检查优化级别（`-O0` 通常比 `-O2` 更"诚实"）。

**5. 硬件问题**

内存地址不存在或未正确初始化（例如外部 SDRAM 未配置时序）。

**调试方法：** 在调试器中查看反汇编，观察启动文件中从 LMA 到 VMA 的复制循环是否正确执行。

**TL;DR：** 初始值不对，90% 是因为启动文件的 `.data` 复制或 `.bss` 清零环节出了问题，先检查启动代码和链接脚本。

## Topic 2：PHY 硬件

### Q6：PHY 和 MAC 有什么区别？

**答：** 一句话：**MAC 负责"组帧/解帧"，PHY 负责"发比特/收比特"**。

| 层面     | MAC（媒体访问控制器）              | PHY（物理层收发器）              |
|----------|-----------------------------------|----------------------------------|
| 位置     | 通常在 MCU 内部                   | 通常是独立芯片（如 LAN8720、DP83848） |
| 职责     | 封装 / 解析以太网帧、CRC 校验、CSMA/CD | 编码/解码、时钟恢复、电平转换    |
| 数字/模拟 | 纯数字逻辑                        | 混合信号（含模拟电路）            |
| 接口     | 通过 MII/RMII 连接到 PHY          | 通过 MDI（差分对）连接到 RJ45     |
| 标准     | IEEE 802.3 第 2 层                | IEEE 802.3 第 1 层               |

在典型系统中，数据流向如下：

```
MCU (内置 MAC) ---MII/RMII---> PHY 芯片 ---MDI (差分信号)---> RJ45 连接器
```

MAC 发送一个帧时，会加上前导码（Preamble）、帧起始定界符（SFD）、目标 MAC、源 MAC、长度/类型字段、数据、填充和 FCS（CRC32）。然后逐比特交给 PHY 发送。

PHY 负责将比特流编码为适合在物理介质上传输的信号（如 100BASE-TX 使用 MLT-3 编码），同时负责时钟恢复和链路检测。

**TL;DR：** MAC 是"软件层的大脑"，处理帧；PHY 是"硬件层的手脚"，处理比特和物理信号。

### Q7：LAN8720 需要外部晶振吗？

**答：** 视时钟方案而定，有三种可能：

**方案 A：外部 50MHz 晶振（不一定需要）**

LAN8720 内部有振荡器电路，可以接 50MHz 晶振（两个电容接地）。但更常见的是下面方案 B。

**方案 B：从 MCU 提供 50MHz 时钟（推荐）**

如果 MCU 有 50MHz 时钟输出能力，可以直接将 MCO（例如 STM32 的 PA8）连接到 LAN8720 的 XTAL1/CLKIN 引脚。此时不需要外部晶振，只需要一个**串联 22Ω 电阻**来抑制反射。

**方案 C：外部有源振荡器**

直接采购 50MHz 有源振荡器模块供电。

**关键点：** RMII 模式下，LAN8720 要求 50MHz 参考时钟 ±50ppm。测量方法：用示波器或频率计测 XTAL1/CLKIN 引脚，确认频率为 50.000MHz ± 2.5kHz。

**常见错误：** 使用了 25MHz 晶振——LAN8720 内部没有 2 倍频 PLL，必须外部提供 50MHz。

**TL;DR：** LAN8720 需要 50MHz 时钟源，可以用外部晶振、MCU 提供或有源振荡器，但必须准确且稳定。

### Q8：为什么 PHY 需要多路电源供电（3.3V、1.2V）？

**答：** 因为 PHY 内部有不同功能模块，各自需要不同的工作电压。

以 LAN8720 为例：

| 电源引脚     | 电压      | 供电对象                                    |
|-------------|-----------|---------------------------------------------|
| VDDIO       | 3.3V      | MAC 接口（MII/RMII 引脚）、寄存器接口（MDIO） |
| VDDA        | 3.3V      | 内部模拟电路（PLL、时钟恢复）                |
| VDDCR       | 1.2V (内部 LDO 产生) | 数字核心逻辑                         |

DP83848 则明确需要外部提供 3.3V（I/O）和 1.8V/2.5V（核心）。

**为什么不能统一用 3.3V？** 数字核心电路使用更先进的工艺节点，工作电压更低（1.2V/1.8V），这样可以降低功耗和发热。如果强行用 3.3V 会给核心供电，芯片会永久损坏。

**PCB 设计时需要注意：** 不同电源域的去耦电容要靠近对应引脚摆放，通常用 0.1μF + 10μF 的组合。模拟电源 VDDA 建议使用 π 型滤波器（磁珠 + 电容）隔离数字噪声。

**TL;DR：** PHY 是多电压器件：I/O 用 3.3V 与 MAC 通信，核心用 1.2V/1.8V 降低功耗，模拟部分用独立电源减少噪声。

### Q9：MII 和 RMII 有什么区别？

**答：** MII（Media Independent Interface）和 RMII（Reduced MII）是 MAC 与 PHY 之间的标准接口。区别在于**引脚数量和时钟频率**。

| 特性              | MII                           | RMII                         |
|-------------------|-------------------------------|------------------------------|
| 数据引脚          | TXD[3:0] + RXD[3:0] = 8 根    | TXD[1:0] + RXD[1:0] = 4 根   |
| 时钟线            | TX_CLK + RX_CLK = 2 根        | REF_CLK = 1 根               |
| 控制线            | TX_EN, RX_DV, TX_ER, RX_ER, CRS, COL | TX_EN, CRS_DV               |
| 总引脚数          | 约 16-18 根                   | 约 7-9 根                    |
| 时钟频率（100M）  | TX_CLK = 25MHz, RX_CLK = 25MHz | REF_CLK = 50MHz              |
| 时钟频率（10M）   | TX_CLK = 2.5MHz, RX_CLK = 2.5MHz | REF_CLK = 50MHz（不变）    |

**为什么 RMII 用更少引脚？**

- MII 用 4 位宽数据总线在 25MHz 下传输 100Mbps（4×25=100）。
- RMII 用 2 位宽数据总线在 50MHz 下传输同样的 100Mbps（2×50=100）。
- RMII 使用单一的 50MHz 参考时钟，而非每个方向各一个时钟。

**设计建议：** 引脚紧张的 MCU（如 LQFP-48 封装）优先选 RMII；引脚充足的系统用 MII 延迟更小。

**TL;DR：** RMII 用 2 倍时钟频率换取了减半的引脚数，节约 PCB 面积，但要求 50MHz 参考时钟。

---

### Q10：为什么 RMII 需要 50MHz 参考时钟？

**答：** 根本原因在于 RMII 用**2 倍频率 × 减半数据位宽**的方式维持了 100Mbps 的吞吐量。

定量计算（100BASE-TX）：

```
MII:   4 位 × 25 MHz = 100 Mbps
RMII:  2 位 × 50 MHz = 100 Mbps
```

**时钟的三种提供方式：**

1. **PHY 提供时钟给 MCU**：PHY 从外部晶振或内部振荡器产生 50MHz，通过 REF_CLK 输出给 MCU。MCU 将此引脚配置为输入。
2. **MCU 提供时钟给 PHY**：MCU 用 MCO 或其他时钟输出引脚产生 50MHz 给 PHY。MCU 将此引脚配置为输出。
3. **外部有源振荡器**：同时给 MCU 和 PHY 提供 50MHz。

**关键时序要求：** RMII 的 TXD、TX_EN 在 REF_CLK 上升沿被采样。时钟抖动（Jitter）要求通常在 ±500ps 以内。如果使用 MCU 的 MCO 输出，要确认 MCO 的抖动指标是否满足 PHY 的要求。

**TL;DR：** RMII 用 50MHz 时钟是因为 2 位数据总线 × 50MHz = 100Mbps，与 4 位 × 25MHz 等效。

### Q11：PHY 可以直接连接 RJ45 而不加网络变压器吗？

**答：** **不能**，这会损坏设备且违反 IEEE 802.3 标准。

**网络变压器（如 H1102、YT48-01）的核心作用：**

1. **电气隔离**：提供 1500V 或更高的隔离电压，防止地环路和浪涌损坏设备。雷电浪涌或地电位差会直接烧毁 PHY 芯片。
2. **共模抑制**：变压器中心抽头通过终端电阻和电容接地，可以有效抑制共模噪声。
3. **电平匹配**：将 PHY 的差分信号转换为标准 MDI 电平。
4. **直流隔离**：阻止直流分量，只允许交流信号通过。

**不接变压器的后果：**

- PHY 芯片几乎立刻损坏（地环流或静电放电）。
- 即使侥幸未立即损坏，长期可靠性极差。
- 无法通过 EMI/EMC 认证。

**带变压器的 RJ45（带内部集成变压器）** 是更简洁的方案：

```
PHY TX+ ---> 变压器（集成在 RJ45 座内） ---> RJ45 引脚 1 (TX+)
PHY TX- ---> 变压器（集成在 RJ45 座内） ---> RJ45 引脚 2 (TX-)
PHY RX+ ---> 变压器（集成在 RJ45 座内） ---> RJ45 引脚 3 (RX+)
PHY RX- ---> 变压器（集成在 RJ45 座内） ---> RJ45 引脚 6 (RX-)
```

中心抽头接法（取决于 PHY 规格）：
- 接 3.3V（LAN8720）：需要输出共模电压。
- 通过 0.1μF 电容接地（DP83848）：需要交流耦合。

**TL;DR：** 网络变压器必须加，它提供隔离、抑制共模噪音、保护 PHY 不被雷击或地环路损坏。

## Topic 3：PHY 软件 / 驱动

### Q12：MDIO 通信失败，我读到 0xFFFF —— 哪里出了问题？

**答：** 通过 MDIO 读取 PHY 寄存器返回 0xFFFF，通常不是 PHY 本身的问题，而是**接口通信失败**。

**排查步骤（按顺序）：**

**1. 确认 PHY 地址**

每个 PHY 有一个 5 位地址（通常由 ADDR 引脚的电平决定）。例如 LAN8720：

| ADDR (pin) | PHY 地址 |
|------------|----------|
| 悬空 / 拉低 | 0x00     |
| 拉高       | 0x01     |

最常用地址为 `0x00`（如果只有一颗 PHY）。确认 `(addr << 2)` 和 `(reg << 0)` 组合正确。

```
// MDIO 帧格式示例（读操作）
PRE(32bits) + ST(01) + OP(10) + PHY_ADDR(5bit) + REG_ADDR(5bit) + TA(20ns turnaround) + DATA(16bit)
```

**2. 确认 MDC 时钟频率**

MDC 最大频率通常为 2.5MHz（IEEE 802.3 规范）。如果 MCU 的 MDC 频率太高（如超过 25MHz），PHY 无法响应，数据线保持高阻态 → 上拉电阻拉高 → 读到 0xFFFF。

**3. 检查上拉电阻**

MDIO 是双向开漏引脚，需要 **1.5kΩ ~ 10kΩ 上拉电阻**到 3.3V。没有上拉电阻时，读操作返回高阻态 → 读到 0xFFFF。

**4. 确认 PHY 已上电且复位完成**

检查 PHY 的 RST 引脚电平。LAN8720 要求复位低电平至少保持 1ms。用示波器或代码延时检查。

**5. 验证 GPIO 复用功能**

MCU 的 MDC/MDIO 引脚必须正确配置为 Alternate Function。误配为 GPIO 推挽输出会导致通信失败。

**TL;DR：** 读到 0xFFFF 几乎总是 MDIO 不通 —— 检查 PHY 地址、MDC 频率、上拉电阻和 GPIO 复用配置。

### Q13：为什么需要读取 BMSR（Register 1）两次才能获得正确的链路状态？

**答：** 这是 IEEE 802.3 规范中定义的自清除（Self-Clearing）行为。

**BMSR（Basic Mode Status Register，地址 0x01）中各比特的含义：**

| Bit | 名称                 | 说明                           |
|-----|----------------------|--------------------------------|
| 15  | 100BASE-T4 能力       |                                |
| 14  | 100BASE-TX 全双工     |                                |
| 13  | 100BASE-TX 半双工     |                                |
| 12  | 10BASE-T 全双工       |                                |
| 11  | 10BASE-T 半双工       |                                |
| 10  | 100BASE-T2 全双工     |                                |
| 9   | 100BASE-T2 半双工     |                                |
| 8   | 扩展状态位             |                                |
| 7   | 扩展能力位             |                                |
| 6   | **无应答（MF）**       | **自清除** — 自动协商失败时置位   |
| 5   | **链路状态已更改**     | **自清除** — 链路状态变化时置位   |
| 4   | **远程故障**           | **自清除**                     |
| 3   | **自动协商完成**       | **自清除**                     |
| 2   | 链路状态               | 只读，1=链路建立                |
| 1   | 链路脉冲检测能力       |                                |
| 0   | 扩展能力               |                                |

**自清除寄存器**（Bit 5、4、3）在每次读取后自动清零。因此：

```
第一次读 BMSR → 看到 Bit 5 (Link Changed)=1，Bit 3 (Auto-Neg Complete)=1
    ↓
这些位被清零
第二次读 BMSR → 看到真实链路状态 Bit 2=1，且变化标志位已清除
```

正确的轮询模式：

```c
uint16_t read_bmsr(void) {
    uint16_t temp = phy_read(PHY_ADDR, PHY_BMSR);  // 第一次：读取并清除自清除位
    return phy_read(PHY_ADDR, PHY_BMSR);            // 第二次：获取当前真实状态
}
```

**TL;DR：** BMSR 中的状态变化位是自清除的，第一次读取会清除原状态，第二次读取才能获得当前真实值。

---

### Q14：自动协商（Auto-Negotiation）和强制模式 —— 用哪个？

**答：** 绝大多数场景使用**自动协商**。但在某些特殊场景下，强制模式是必要的。

**自动协商（推荐）**

- AN 双方通过 Fast Link Pulses（FLP）交换能力码（Base Page + Next Pages）。
- 自动选择双方都支持的**最高速度 × 最佳双工模式**。
- 优先级从高到低：100BASE-TX 全双工 > 100BASE-TX 半双工 > 10BASE-T 全双工 > 10BASE-T 半双工。

```
自动协商后读取 ANLPAR（Auto-Neg Link Partner Ability，地址 0x05）：
0x05E1 → 对端支持 100BASE-TX 全双工 + 100BASE-TX 半双工 + 10BASE-T 全双工 + 10BASE-T 半双工
```

**强制模式**

- 直接设置 PHY 的速度和双工模式，不进行 AN。
- 需要手动设置以下几个寄存器：

```c
// LAN8720 强制 100M 全双工示例
uint16_t control = phy_read(PHY_ADDR, PHY_BCR);  // 地址 0x00
control |= (1 << 13);   // Bit 13: 速度选择，1=100Mbps
control |= (1 << 8);    // Bit 8: 双工模式，1=全双工
control &= ~(1 << 12);  // Bit 12: 关闭自动协商
phy_write(PHY_ADDR, PHY_BCR, control);
// 然后软件复位使配置生效
```

**何时用强制模式？**

- 对端设备不支持自动协商（老式工业设备）。
- 需要精确控制协商结果。
- 在同一个交换机端口上只能有一个 Master，其他需设为 Slave（如 1000BASE-T）。

**何时用自动协商？** 几乎所有通用嵌入式以太网系统。不匹配速率（100M vs 10M）或双工（全双工 vs 半双工）会导致大量 CRC 错误。

**TL;DR：** 除非连接的老旧设备不支持，否则永远使用自动协商——它可以避免速率和双工不匹配的问题。

---

### Q15：自动协商需要多长时间？

**答：** 通常需要 **1.5 ~ 3 秒**，理论上限约 4 秒。

**时间线分解：**

| 阶段                         | 时间            | 说明                          |
|------------------------------|----------------|-------------------------------|
| 链路脉冲检测                  | 0 ~ 300ms      | 检测到对端链路脉冲              |
| FLP 突发交换（Base Page）    | 50 ~ 200ms     | 交换能力码                     |
| Next Page 交换（可选）        | 0 ~ 2.4s       | 交换更多能力信息（通常不需要）   |
| 协商完成 + 链路稳定           | 0 ~ 1s         | PHY 锁定速度/双工，链路建起     |

**如何检测协商完成：**

```c
// 方法 1：轮询 BMSR（Register 1）Bit 3（Auto-Neg Complete）
while (1) {
    uint16_t bmsr = phy_read(PHY_ADDR, PHY_BMSR);
    bmsr = phy_read(PHY_ADDR, PHY_BMSR);  // 读取两次
    if (bmsr & (1 << 3)) {  // Bit 3 = AN Complete
        break;
    }
    delay_ms(10);
}

// 方法 2：轮询连接状态
while (!(phy_read(PHY_ADDR, PHY_BMSR) & (1 << 2))) {  // Bit 2 = Link Status
    delay_ms(10);
}
```

**常见问题：** 设备插拔后重新协商，在 Linux 中 `ifconfig eth0 down / up` 会触发新的协商过程，必须等待 2~3 秒才能收发数据。

**TL;DR：** 自动协商通常需要 2~3 秒，代码中需要等待协商完成标志后再开始网络通信。

### Q16：链路状态变化为什么会导致 MCU 死机？

**答：** 这不是 PHY 本身的问题，而是**中断处理不当**导致的。

**典型场景：**

```c
// PHY 中断处理函数 —— 错误示范
void PHY_IRQHandler(void) {
    uint16_t status = phy_read(PHY_ADDR, PHY_STATUS);
    if (status & LINK_DOWN) {
        // 做一些耗时操作
        delay_ms(100);        // 在中断里延时 —— 危险！
        printf("Link Down\n"); // 使用串口输出 —— 可能阻塞！
        TCP_reset_connection(); // 复杂操作 —— 可能导致递归或死锁！
    }
    phy_clear_interrupt(PHY_ADDR);
}
```

**为什么死机：**

1. **中断优先级不当**：如果 PHY 中断优先级高于 SysTick，延时函数无法工作。
2. **在中断中调用不可重入函数**：`printf`、`malloc` 等函数可能不可重入。
3. **中断嵌套导致栈溢出**：中断处理中调用了涉及其他中断的函数。
4. **TCP 协议栈状态机混乱**：在中断上下文中操作协议栈内部状态。

**正确做法 —— 中断只做标记：**

```c
volatile uint8_t phy_link_changed = 0;

void PHY_IRQHandler(void) {
    phy_link_changed = 1;  // 仅置标志
    phy_clear_interrupt(PHY_ADDR);
    // 立即退出中断
}

void main_loop(void) {
    while (1) {
        if (phy_link_changed) {
            phy_link_changed = 0;
            uint16_t bmsr = phy_read(PHY_ADDR, PHY_BMSR);
            bmsr = phy_read(PHY_ADDR, PHY_BMSR);  // 读两次
            if (bmsr & (1 << 2)) {
                printf("Link UP\n");
                TCP_reconnect();
            } else {
                printf("Link DOWN\n");
                TCP_close();
            }
        }
        // 其他任务...
    }
}
```

**TL;DR：** PHY 中断里不要做耗时操作，只需设一个标志位，在主循环中处理链路变化。

### Q17：PHY 软件复位到底做了什么？

**答：** PHY 软件复位是通过写 BCR（Basic Control Register，地址 0x00）的 Bit 15 触发的。

```c
// 软件复位方法
uint16_t bcr = phy_read(PHY_ADDR, PHY_BCR);
bcr |= (1 << 15);           // Bit 15: Reset
phy_write(PHY_ADDR, PHY_BCR, bcr);

// 等待复位完成（Bit 15 自动清零）
while (phy_read(PHY_ADDR, PHY_BCR) & (1 << 15)) {
    delay_ms(1);
    if (timeout++ > 100) break;  // 超时保护
}
```

**复位完成的内容：**

| 寄存器/功能         | 复位值                    |
|---------------------|---------------------------|
| BCR (0x00)          | 0x3100（自动协商使能）     |
| BMSR (0x01)         | 读取后可见                 |
| PHYID1 (0x02)       | 厂商 ID（只读，不变）       |
| PHYID2 (0x03)       | 型号/版本（只读，不变）     |
| ANAR (0x04)         | 0x01E1（默认能力）         |
| 所有控制/状态寄存器  | 恢复默认值                |
| 链路状态            | 断开，重新开始自动协商     |

**复位不会做的：**

- 不会修改 PHY 地址（由硬件引脚决定）。
- 不会改变芯片内部固件版本。
- 不会清除 MDIO/MDC 接口设置。

**复位后的等待时间：** LAN8720 软件复位需要约 1ms，DP83848 需要约 800μs。复位完成后建议额外等待 10ms再开始 MDIO 通信。

**TL;DR：** 软件复位将 PHY 所有寄存器恢复到默认值（连线断开、重新协商），相当于 PHY 重新上电但不影响硬件引脚配置。

## Topic 4：网络协议

### Q18：以太网最小帧长度是多少？为什么是 64 字节？

**答：** 标准以太网帧的最小长度为 **64 字节**（从目标 MAC 地址到 FCS 结尾）。

**以太网帧结构：**

| 字段             | 长度（字节） |
|------------------|-------------|
| 前导码（Preamble）| 7           |
| SFD              | 1           |
| 目标 MAC         | 6           |
| 源 MAC           | 6           |
| 长度/类型        | 2           |
| 数据（Payload）  | **46 ~ 1500** |
| FCS (CRC32)      | 4           |
| **帧间隔（IFG）** | 12          |

**64 字节的由来：**

```
(目标 MAC 6 + 源 MAC 6 + 长度/类型 2 + 数据 46 + FCS 4) = 64 字节
```

**为什么有这个限制？**

这与 CSMA/CD（载波监听多路访问/冲突检测）有关。在 10Mbps 半双工模式下，最远距离的两个站点之间，数据往返时间（RTT）必须小于最短帧的发送时间，才能保证发送方在发送完毕前检测到冲突。

```
最小帧长 64 字节 × 8 位/字节 = 512 位
512 位 / 10 Mbps = 51.2 μs  ← 这定义了最大冲突域距离
```

在 100Mbps 下，512 位时间 = 5.12μs，因此 100BASE-TX 的最大网络直径比 10BASE-T 小得多。

**TL;DR：** 64 字节最小帧由 CSMA/CD 的冲突检测时间决定，数据不足 46 字节时需填充（Padding）到 46 字节。

---

### Q19：如果我发送少于 64 字节的数据包会发生什么？

**答：** 这个帧会被认为是**碎片帧（Runt Frame）**，接收方将丢弃它。

**场景举例：** 发送 10 字节数据（加上帧头 14 + FCS 4 = 28 字节，小于 64）。

**在发送端（你的 MCU）：**

- 如果 MAC 支持自动填充（大多数 MCU 的 MAC 都支持），MAC 硬件会自动在数据后面填充 0x00 直到满足 46 字节的最小数据长度。
- 如果不支持自动填充，MAC 会发送一个短帧（Runt）。

**在接收端（交换机/PC）：**

- 交换机收到小于 64 字节的帧，直接丢弃。
- 如果交换机的 "Receive Error" 计数器增加，通常是收到了 Runt 或 CRC 错误的帧。
- Wireshark 会标记为 **"Short frame"** 或 **"Packet size limited during capture"**。

**在 MAC 层配置自动填充：**

```c
// STM32 ETH MAC 配置示例
ETH_InitStructure.ETH_TxPreambleLength = ETH_TXPREAMBLELENGTH_7BYTES;
// 自动填充使能
ETH_InitStructure.ETH_AutomaticPadCRCStrip = ENABLE;  // 接收侧剥除填充
// 或
ETH->MACCFGR |= MAC_CR_IPFCO;  // 启用填充和 CRC 自动处理
```

**手动填充方法（不推荐，但了解一下原理）：**

```c
uint8_t frame[64];
uint8_t data_len = 10;

// 构建以太网帧头
memcpy(&frame[0], dest_mac, 6);
memcpy(&frame[6], src_mac, 6);
frame[12] = (type >> 8) & 0xFF;
frame[13] = type & 0xFF;

// 填充数据
memcpy(&frame[14], user_data, data_len);

// 如果数据不足 46 字节，填充 0x00
uint8_t pad_len = (data_len < 46) ? (46 - data_len) : 0;
if (pad_len > 0) {
    memset(&frame[14 + data_len], 0x00, pad_len);
}

// 计算并追加 CRC32（FCS）
uint32_t crc = calculate_crc32(frame, 14 + 46);
// 注意：最终发送的总长度为 14 + 46 + 4 = 64 字节
```

**TL;DR：** 少于 64 字节的帧是 Runt Frame，会被接收方丢弃。MAC 硬件通常会帮你自动填充到 46 字节数据。

### Q20：ARP 如何知道某个应答对应的是我的请求？

**答：** 通过 **Transaction ID（操作 ID / 请求标识）**，但在标准 ARP 中，它实际上是通过**双层匹配**实现的。

**标准 ARP（RFC 826）的帧结构：**

| 字段               | 长度    | 说明              |
|--------------------|---------|-------------------|
| 硬件类型           | 2 字节  | 1 = Ethernet      |
| 协议类型           | 2 字节  | 0x0800 = IPv4     |
| 硬件地址长度       | 1 字节  | 6（MAC 地址长度）  |
| 协议地址长度       | 1 字节  | 4（IPv4 地址长度） |
| 操作码             | 2 字节  | 1=请求, 2=应答    |
| 发送方 MAC         | 6 字节  |                   |
| 发送方 IP          | 4 字节  |                   |
| 目标 MAC           | 6 字节  | 请求时为 00:00:00:00:00:00 |
| 目标 IP            | 4 字节  |                   |

**ARP 的匹配逻辑**靠以下两层保证：

1. **以太网帧头匹配**：目标 MAC 字段必须是本机 MAC。ARP 应答的以太网帧头中，目标 MAC 是请求方的 MAC 地址。
2. **ARP 包内字段匹配**：主机 A 收到 ARP 应答后，验证 ARP 包中的 **目标 IP 地址是否等于自己的 IP**，以及**发送方 IP 是否等于自己请求的 IP**。

**注意：ARP 没有像 ICMP Echo Request 那样的 ID 号**。它完全是靠"我请求了 192.168.1.1，如果收到一个包说 192.168.1.1 的 MAC 是 xx:xx:xx:xx:xx:xx 且目标 MAC 是我，那就是给我的应答"。

**ARP 缓存表（ARP Table）:**

当主机 A 收到 ARP 应答时，它会把 `发送方 IP <-> 发送方 MAC` 这对映射存入 ARP 缓存中，供后续使用。

```
主机 A 的 ARP 表（通过 `arp -a` 查看）：
192.168.1.1      → 00:11:22:33:44:55   (动态，过期时间 120 秒)
192.168.1.100    → aa:bb:cc:dd:ee:ff   (静态)
```

**TL;DR：** ARP 没有 ID 号，通过"目标 MAC=我 + ARP 包内目标 IP=我"双重匹配来确定应答属于我的请求。

### Q21：为什么 TCP 需要三次握手？

**答：** 三次握手（Three-Way Handshake）的目的：**同步双方的序列号（SEQ）并确认连接可用**，防止历史连接干扰和资源浪费。

**过程详解：**

```
客户端 (Client)                    服务器 (Server)
    |                                   |
    |  ==== SYN (SEQ=x) ===========>    |
    |                                   |
    |  <======= SYN+ACK (SEQ=y,  =====  |
    |            ACK=x+1)               |
    |                                   |
    |  ====== ACK (SEQ=x+1, ========>   |
    |         ACK=y+1)                  |
    |                                   |
    |  <======== 数据传输开始 ========>  |
```

**为什么恰好是三次，不是两次或四次？**

**如果只有两次握手：**

- 客户端发送的 SYN 在网络中延迟，重传后建立了连接，数据传输完毕关闭连接。
- 此时延迟的旧 SYN 到达服务器，服务器以为是新连接请求，分配资源等待。
- 客户端不理会这个旧的 SYN+ACK，但服务器会一直等待，造成资源浪费。

**三次握手的核心功能：**

| 握手次数 | 作用                                    |
|---------|-----------------------------------------|
| 第 1 次  | 客户端发送初始 SEQ 并告知窗口大小         |
| 第 2 次  | 服务器确认收到客户端的 SEQ，发送自己的 SEQ |
| 第 3 次  | 客户端确认收到服务器的 SEQ，连接建立       |

第三次握手还可以携带数据（TCP Fast Open），减少 RTT。

**TL;DR：** 三次握手同步双方的序列号，防止历史连接干扰（旧 SYN 造成资源浪费），同时确认双向通信可达。

### Q22：什么是 VLAN 标签（VLAN Tag），什么时候需要用它？

**答：** VLAN 标签是 IEEE 802.1Q 定义的一个**4 字节插入字段**，用于在网络中隔离不同的逻辑网络。

**VLAN 帧结构：**

```
标准以太网帧：
| 目标 MAC | 源 MAC | 类型/长度 | 数据 | FCS |
|    6     |   6    |    2      | 46-1500 | 4  |

带 VLAN 标签的帧：
| 目标 MAC | 源 MAC | TPID | TCI | 类型/长度 | 数据 | FCS |
|    6     |   6    |  2   |  2  |    2      | 46-1500 | 4  |
```

| 字段     | 长度    | 说明                                    |
|----------|---------|----------------------------------------|
| TPID     | 2 字节  | 0x8100（VLAN 协议标识）                 |
| TCI      | 2 字节  | 其中 VID 占 12 位（0~4095），PCP 占 3 位（优先级），DEI 占 1 位 |

**典型应用场景：**

| 场景                          | 说明                                       |
|-------------------------------|--------------------------------------------|
| 工厂网络隔离                  | 同一个交换机，VLAN 10=PLC 控制网，VLAN 20=办公网 |
| 语音和数据分离                | VLAN 100=IP 电话，VLAN 200=PC 数据          |
| 安全隔离                      | 访客 VLAN 不能访问内部 VLAN                 |

**MCU 开发者什么时候需要考虑 VLAN？**

- MCU 直连交换机且交换机端口配置为 Trunk（带 Tag 的端口）。
- MCU 内部运行了 VLAN 感知的应用（如网管型交换机控制器）。
- MCU 作为边缘设备，通常连 Access 端口（不带 Tag），不需要关心 VLAN。

**TL;DR：** VLAN 标签在帧头增加 4 字节 (0x8100 + 12 位 VLAN ID)，用于逻辑隔离不同设备组。嵌入式设备通常连 Access 端口，无需处理 VLAN 标签。

### Q23：我的设备通过 DHCP 获取不到 IP —— 可能是什么问题？

**答：** DHCP（Dynamic Host Configuration Protocol）是一个四步过程（DORA），排查需要逐层验证。

**DHCP 流程（DORA）：**

```
客户端                    DHCP 服务器
   |                          |
   |  ==== DISCOVER =======>  |  (广播，源 IP=0.0.0.0，目标 IP=255.255.255.255)
   |                          |
   |  <==== OFFER =========== |  (通常也是广播，提供 IP、子网掩码、网关、DNS)
   |                          |
   |  ==== REQUEST =========> |  (广播，请求确认使用该 IP)
   |                          |
   |  <==== ACK ============= |  (确认，租约开始)
```

**排查清单：**

| 现象                         | 可能原因                           | 验证方法                            |
|------------------------------|------------------------------------|-------------------------------------|
| 没有收到 DISCOVER 的应答      | 链路未建立                          | 检查 Link LED / BMSR Bit 2          |
|                              | DHCP 服务器不在同一广播域           | 手配 IP 能否 ping 通网关            |
|                              | 交换机端口配置了 DHCP Snooping 未信任 | 检查交换机端口配置                  |
| OFFER 收到但 REQUEST 被忽略  | 客户端 MAC 被 DHCP 服务器拉黑       | 检查 DHCP 服务器日志                 |
|                              | 地址池已满                          | 检查 DHCP 服务器地址池               |
| ACK 收到但无法上网            | 网关地址错误或不在同一子网           | `ping` 网关地址                      |
|                              | DNS 服务器不可达                    | `ping` 8.8.8.8                       |
| DHCP 工作但 50% 租约时间后断网| 租约未续期                          | 检查 DHCP 客户端是否实现了续期逻辑   |

**Wireshark 抓包验证：**

```
No.  Time    Source          Destination     Protocol  Info
1   0.000   0.0.0.0         255.255.255.255 DHCP      DHCP Discover
2   0.015   192.168.1.1     255.255.255.255 DHCP      DHCP Offer (192.168.1.100)
3   0.020   0.0.0.0         255.255.255.255 DHCP      DHCP Request (192.168.1.100)
4   0.030   192.168.1.1     255.255.255.255 DHCP      DHCP ACK (192.168.1.100)
```

**开源方案：** lwIP 的 DHCP 实现较成熟，注意在回调函数中保存获取到的 IP、网关、DNS：

```c
static void dhcp_callback(const struct dhcp_state *state, void *arg) {
    if (state->state == DHCP_STATE_BOUND) {
        ip4_addr_t ip = state->offered_ip_addr;
        ip4_addr_t gw = state->offered_gw;
        ip4_addr_t dns = state->offered_dns;
        printf("DHCP Bound: IP %s\r\n", ip4addr_ntoa(&ip));
    }
}
```

**TL;DR：** DHCP 不成功，先用 Wireshark 抓包确认 DORA 四步走到了哪一步，链路层是最常被忽略的瓶颈。

## Topic 5：交换机

### Q24：集线器（Hub）、交换机（Switch）和路由器（Router）有什么区别？

**答：** 三者在 OSI 模型中所处层级不同，核心功能不同。

| 设备       | OSI 层次          | 工作原理                     | 转发依据        | 隔离冲突域？ | 隔离广播域？ |
|------------|-------------------|------------------------------|----------------|-------------|-------------|
| 集线器 Hub  | 第 1 层（物理层）  | 简单中继，所有端口共享带宽     | 无（电气复制）  | 否          | 否          |
| 交换机 Switch | 第 2 层（数据链路层） | 存储转发，硬件查表           | MAC 地址表     | 是          | 否          |
| 路由器 Router | 第 3 层（网络层）   | 路由协议确定转发路径          | IP 地址表      | 是          | 是          |

**Hub（集线器）：**

- 所有端口在一个冲突域内。如果有两台设备同时发送数据，就会发生冲突。
- 10Mbps Hub 的总带宽被所有端口共享。

```
Hub 的行为：收到来自 A 的数据，向所有其他端口复制一份（Store-and-Forward? 不，是"Replicate-and-Forward"）
```

**Switch（交换机）：**

- 每个端口是独立的冲突域。
- 根据目标 MAC 地址查表，只将帧转发到对应端口。
- 带宽 = 端口数量 × 端口速率（全双工模式下）。

```
Switch 的行为：收到来自 A 发往 B 的帧，查 MAC 表后只在 Port B 发出，其他端口不受影响。
```

**Router（路由器）：**

- 连接不同网段，在 IP 层转发。
- 维护路由表（静态或动态）。
- 可以做 NAT（网络地址转换）、防火墙等。

**TL;DR：** Hub 复制到所有端口（无智能），Switch 根据 MAC 地址只转发到目标端口（智能转发），Router 根据 IP 地址在不同网段间路由。

### Q25：管理型交换机（Managed Switch）和非管理型交换机有什么区别？

**答：** 核心区别在于**能否配置和监控交换机的行为**。

| 特性             | 非管理型交换机                          | 管理型交换机                                    |
|------------------|----------------------------------------|-----------------------------------------------|
| 配置界面          | 无（即插即用）                          | Web / CLI / SNMP / Telnet / SSH               |
| VLAN             | 不支持                                 | 支持 802.1Q VLAN                               |
| QoS              | 无                                     | 支持基于端口/协议的优先级                        |
| 端口镜像          | 无                                     | 支持（把某个端口的数据复制到监控端口）           |
| SNMP 监控        | 无                                     | 支持（远程监控流量、错误计数等）                 |
| 链路聚合（LAG）   | 不支持                                 | 支持（将多个端口捆绑为一个逻辑链路）             |
| 生成树协议（STP） | 通常内置但不能配置                      | 可配置 RSTP / MSTP                              |
| 端口安全          | 无                                     | MAC 地址过滤、风暴控制、DHCP Snooping           |
| 价格              | 便宜（几十元）                          | 较贵（几百到数千元）                             |

**嵌入式开发中的典型场景：**

| 场景                           | 交换机类型     | 原因                                          |
|--------------------------------|---------------|-----------------------------------------------|
| 家用路由器内置交换机             | 非管理型       | 无需配置，只需基本连通                          |
| 工业控制器多端口组网             | 管理型         | 需要 VLAN 隔离、端口优先级、远程监控            |
| 基于 MCU 的交换机设计（如 RT-Thread） | 管理型（定制）  | MCU 通过 MDIO/SPI 控制交换芯片的寄存器          |

**TL;DR：** 非管理型交换机即插即用功能简单；管理型交换机支持 VLAN、端口镜像、QoS 等高级功能，但需要配置。

### Q26：Port 1 和 Port 2 之间不能通信 —— 可能是什么问题？

**答：** 端口间不通有多种可能，按排查顺序列出。

| 排查步骤                        | 验证方法                                              | 解决方案                            |
|--------------------------------|------------------------------------------------------|-------------------------------------|
| 1. 链路层是否建立               | 检查端口 LED / 交换机状态寄存器                         | 更换网线或强制端口速率               |
| 2. 端口是否被禁用               | 检查端口状态寄存器                                     | `switch_port_enable(port, 1)`       |
| 3. 是否在同一个 VLAN            | `show vlan` 查看端口所属 VLAN                          | 分配同一 VLAN 或设置为 Trunk         |
| 4. MAC 地址表是否正常           | `show mac-address-table` 查看学习情况                   | 检查是否有 MAC 冲突                  |
| 5. 生成树协议（STP）阻塞         | `show spanning-tree` 查看端口状态                      | 等待 STP 收敛（30~50秒）或禁用 STP   |
| 6. 端口安全 / MAC 过滤          | `show port-security`                                   | 添加允许的 MAC 或禁用端口安全        |
| 7. 双工 / 速率不匹配            | 检查端口的协商结果                                     | 两端强制设置为相同的速率和双工模式    |

**具体排查命令（CLI 管理型交换机）：**

```
// 查看端口状态
show interfaces status
  Port   Type             Duplex Speed   Neg  Link
  Port1  100BASE-TX       Full   100     Auto Up
  Port2  100BASE-TX       Full   100     Auto Up

// 查看 VLAN 配置
show vlan
  VLAN 1 (Default): Port1, Port3, Port4
  VLAN 10: Port2          // ← Port2 在 VLAN 10，与 Port1 不在同一 VLAN！
```

**MCU 控制交换芯片的场景：**

如果 MCU 通过 SPI/MDIO 控制交换芯片，先读取 Port 状态寄存器确认 link 状态：

```c
// 读取 Port 1 状态寄存器（KSZ8794 示例）
uint16_t port1_status = switch_read_reg(P1_STS);
// Bit 0 = Link Up
// Bit 1 = 100Mbps
// Bit 2 = Full Duplex
```

**TL;DR：** 端口不通先看 Link LED，再看 VLAN 配置，最后检查 STP 状态 —— 80% 的原因是 VLAN 不一致或 STP 未收敛。

### Q27：交换机如何学习 MAC 地址？

**答：** 交换机通过**逆向学习法（Backward Learning）** 自动构建 MAC 地址表。

**学习过程详解：**

**初始状态：** MAC 地址表为空。

**步骤 1：A 发送数据给 B**

```
A (MAC: AA:AA:AA:AA:AA:AA) ← Port 1
                                             交换机 MAC 表（空）
B (MAC: BB:BB:BB:BB:BB:BB) ← Port 2
```

交换机收到帧后：
1. 提取**源 MAC 地址**（AA:AA:AA:AA:AA:AA）。
2. 记录：`AA:AA:AA:AA:AA:AA → Port 1`。
3. 提取**目标 MAC 地址**（BB:BB:BB:BB:BB:BB），查表未找到。
4. **向除 Port 1 外的所有端口泛洪**（Flooding），即转发到 Port 2、Port 3、Port 4...

**步骤 2：B 回包给 A**

```
B 回复 A 时：
1. 提取源 MAC（BB:BB:BB:BB:BB:BB），记录：`BB:BB:BB:BB:BB:BB → Port 2`。
2. 提取目标 MAC（AA:AA:AA:AA:AA:AA），查表找到 → Port 1。
3. 精确转发，只发往 Port 1，不再泛洪。
```

**MAC 地址表结构：**

| MAC 地址                  | 端口   | VLAN | 类型   | 老化时间（秒） |
|---------------------------|--------|------|--------|--------------|
| AA:AA:AA:AA:AA:AA        | Port 1 | 1    | 动态   | 300          |
| BB:BB:BB:BB:BB:BB        | Port 2 | 1    | 动态   | 300          |

**关键特性：**

- **动态学习**：每次收到帧都会更新 MAC 表条目并重置老化计时器。
- **老化机制**：如果某个 MAC 在老化时间内没有新帧（通常 300 秒），该条目被删除。
- **静态条目**：可以手动配置静态 MAC 地址表条目（用于端口安全或固定拓扑）。

**TL;DR：** 交换机通过源 MAC 地址反推端口号来学习，收到新帧就记录（源 MAC, 端口）对，查不到目标 MAC 时就向所有端口泛洪。

## Topic 6：调试

### Q28：没有示波器 —— 如何调试 PHY 问题？

**答：** 没有示波器确实限制很大，但仍然有几种有效的调试方法。

**方法 1：观察 LED 指示灯**

大多数 PHY 评估板或 RJ45 座都带有 LED：

| LED 状态                  | 含义                        |
|--------------------------|-----------------------------|
| Link LED 常亮（绿灯）      | 链路已建立                   |
| Link LED 闪烁             | 有数据活动                   |
| Link LED 不亮             | 无链路连接（检查线缆/对端）   |
| Speed LED 亮（黄灯）       | 100Mbps                     |
| Speed LED 不亮            | 10Mbps                      |
| 双色 LED 交替闪烁          | 冲突或错误                   |

**方法 2：MDIO 寄存器轮询（最实用）**

通过串口终端轮询 PHY 寄存器，分析状态变化：

```c
// 调试助手中调用
void phy_debug(void) {
    uint16_t bcr  = phy_read(PHY_ADDR, 0x00);
    uint16_t bmsr = phy_read(PHY_ADDR, 0x01);
    bmsr = phy_read(PHY_ADDR, 0x01);  // 第二次读取
    uint16_t phyid1 = phy_read(PHY_ADDR, 0x02);
    uint16_t phyid2 = phy_read(PHY_ADDR, 0x03);
    uint16_t anar  = phy_read(PHY_ADDR, 0x04);
    uint16_t anlpar = phy_read(PHY_ADDR, 0x05);

    printf("BCR: 0x%04X\n", bcr);
    printf("BMSR: 0x%04X (Link: %s, AN: %s)\n",
           bmsr,
           (bmsr & 0x0004) ? "UP" : "DOWN",
           (bmsr & 0x0020) ? "Complete" : "In Progress");
    printf("PHYID: 0x%04X 0x%04X\n", phyid1, phyid2);
    printf("ANAR: 0x%04X, ANLPAR: 0x%04X\n", anar, anlpar);
}
```

**方法 3：万用表测量**

| 测量点                     | 正常值            | 异常值判断          |
|---------------------------|------------------|-------------------|
| PHY 供电引脚               | 3.3V ± 5%       | < 3.0V 或为 0V    |
| 核心电压（如 VDDCR）       | 1.2V ± 5%       | 为 0V（LDO 问题）  |
| XTAL1/CLKIN 引脚           | 0.5~1.5V (AC)   | 0V 或 3.3V（无时钟）|
| RST 引脚                   | 3.3V（释放后）    | 0V（持续复位中）   |
| MDIO 引脚                  | 3.3V（空闲时）    | 0V（无上拉）       |
| TX+ / TX- 差分信号          | 有电压差（0.2V+） | 无差（无活动）     |

**方法 4：使用交换机/RJ45 端口计数器**

如果用了管理型交换机，查看端口计数器：

```
show interfaces counters Port1
  Good Frames: 12345
  CRC Errors:  0
  Collisions:  0
  Runt Frames: 0
```

**TL;DR：** 没有示波器时，通过 PHY 寄存器读值 + LED 指示 + 万用表测量电压 + 交换机端口计数器，可以定位 80% 的问题。

### Q29：Wireshark 显示我的设备发送的包有 CRC 错误 —— 是什么原因？

**答：** CRC 错误表示**以太网帧末尾的 FCS（Frame Check Sequence，CRC32）校验与接收方计算的值不匹配**。

**可能原因排名：**

| 原因                    | 概率   | 说明                                              |
|------------------------|--------|---------------------------------------------------|
| 双工模式不匹配          | ★★★★★  | 一侧全双工一侧半双工导致冲突，被损坏的帧仍被转发   |
| PCB 信号完整性差        | ★★★★   | 差分线阻抗不匹配、走线过长、地平面不连续           |
| 时钟抖动过大            | ★★★    | REF_CLK 抖动超过 PHY 规格                          |
| 网线质量问题            | ★★★    | 劣质网线、过长（超过 100m）、接触不良               |
| PHY 与 MAC 速率不匹配   | ★★     | MAC 配置 100M 但 PHY 协商到 10M                    |
| 软件 CRC 计算错误       | ★★     | 手动计算 CRC 时忘记包含填充字节                      |
| 电磁干扰（EMI）         | ★★     | 靠近电机、继电器等强干扰源                          |

**如何定位：**

**双工不匹配是最常被忽视的原因**。例如：

```
MCU（强制 100M 全双工） ←→ 交换机（自动协商到 100M 半双工）
```

此时：
- MCU 发送时不管对方是否在发数据。
- 如果交换机也在发送，发生碰撞（Collision）。
- 被破坏的帧通过，CRC 错误产生。

**排查方法：**

```c
// 确认本机 PHY 协商结果
uint16_t anlpar = phy_read(PHY_ADDR, 0x05);
// Bit 8: 100BASE-TX 全双工, Bit 7: 100BASE-TX 半双工, Bit 6: 10BASE-T 全双工, Bit 5: 10BASE-T 半双工
printf("LP Ability: 0x%04X\n", anlpar);
if (anlpar & 0x0100) printf("对端支持 100M 全双工\n");
if (!(anlpar & 0x0100) && (anlpar & 0x0080)) printf("匹配为 100M 半双工！\n");
```

**关于 CRC32：** 以太网 FCS 使用 IEEE 802.3 CRC32（多项式 `0x04C11DB7`），计算范围包括目标 MAC、源 MAC、类型/长度、数据（含填充）。不包含前导码和 SFD。

**TL;DR：** CRC 错误的第一大原因是双工不匹配（一侧全双工一侧半双工），其次才是硬件信号质量问题。

### Q30：我的设备能发送数据包，但没有人回应 —— 如何调试？

**答：** 这是"能讲不能听"的典型问题，说明**接收路径**有问题。

**排查步骤：**

**1. 确认 PHY 接收是否正常**

```c
// 检查 BMSR 的 Link 位
uint16_t status = phy_read(PHY_ADDR, PHY_BMSR);
status = phy_read(PHY_ADDR, PHY_BMSR);  // 读两次
if (!(status & 0x0004)) {
    printf("链路未建立！\n");
    // 检查网线和对端设备
}
```

**2. 检查 RMII/MII 接收引脚**

```
PHY RX+ ---> MCU RMII_RXD0
PHY RX- ---> MCU RMII_RXD1
PHY RX_DV ---> MCU RMII_CRS_DV
```

- RX_DV 信号是否有电平变化（用万用表/逻辑分析仪）？
- RX_DV 在没有数据时应为低电平，有数据时为高电平。

**3. 检查 MCU MAC 的接收配置**

```c
// STM32 MAC 配置示例 —— 检查以下位是否使能
ETH->MACCR |= MAC_CR_RE;        // Bit 2: Receiver Enable
ETH->MACFFR |= MAC_FFR_RA;      // Bit 31: Receive All（用于调试时接收所有包）
// 或者设置正确的 MAC 地址过滤
ETH->MACA0HR = (mac[5] << 8) | mac[4];  // MAC 地址高位
ETH->MACA0LR = (mac[3] << 24) | (mac[2] << 16) | (mac[1] << 8) | mac[0];  // 低位
```

**4. 检查 DMA 描述符（如果使用了 DMA）**

许多 MCU 的以太网 MAC 使用 DMA 传输帧。DMA 描述符的状态必须正确：

```c
// lwIP + STM32 的 DMA 描述符状态检查
struct eth_dma_rx_descriptor *desc = rx_desc_tab;
if (desc->Status & ETH_DMARXDESC_OWN) {
    printf("DMA 描述符仍被 DMA 拥有，CPU 无法访问\n");
}
// 检查接收状态位
if (desc->Status & ETH_DMARXDESC_ES) {
    printf("接收错误：0x%08X\n", desc->Status);
}
```

**5. Wireshark 抓包对比**

在没有交换机端口镜像的情况下，可以将 PC 和设备同时连接到同一台交换机，在 PC 上用 Wireshark：

```
// 在 PC 上抓 Wireshark：
// 如果能看到设备发出的 ARP/DHCP 包，说明发送路径 OK
// 如果看不到 PC 发出的 ARP 应答到达设备，说明接收路径有问题
```

同时在该 Wireshark 中检查：
- 是否有来自设备的包？→ 发送路径正常。
- 是否有发给设备的包？→ 如果中间有交换机但交换机发送了但没有收到，说明接收路径有单向问题。
- 包的 CRC 是否正常？→ 见 Q29。

**6. 硬件排查（最后手段）**

用示波器检查 PHY 的接收差分对（RX+ / RX-）和 RX_DV / CRS_DV 信号。如果没有示波器，用逻辑分析仪看 RX_DV 和 RXD[1:0] 的同步关系。

**TL;DR：** "能发不能收"通常不是发送的问题，而是接收路径的某个环节：PHY 接收、MAC 接收配置、DMA 描述符或地址过滤。用 Wireshark 确认设备发的包正常到达交换机/PC，然后逐级检查接收链路。
